# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Runtime

Use Bun everywhere. Never use `npm`, `yarn`, `pnpm`, `node`, `ts-node`, `npx`, `jest`, `vitest`, or `vite`.

- `bun install` — install deps
- `bun run <script>` — run package.json scripts
- `bunx <pkg>` — instead of npx
- Bun auto-loads `.env.local`, so never use `dotenv` in app code (drizzle.config.ts is the exception — it calls `config()` because drizzle-kit runs outside Next.js).

## Commands

Run from `apps/web/`:

```sh
bun run dev        # Next.js dev server on :3015
bun run build      # production build
bun run test       # integration tests — requires DATABASE_URL, hits real Neon DB
bun run db:push    # push Drizzle schema to Neon (requires DATABASE_URL)
bun run db:generate # regenerate Better Auth schema (lib/auth-schema.ts)
```

Run from repo root (Turborepo):

```sh
bun run dev        # runs all apps in parallel
bun run build
```

## Architecture

This is a Turborepo monorepo with one app: `apps/web` — a Next.js 15 / React 19 marketing site for Burn, a macOS menu bar app that shows Claude usage.

### Data flow: waitlist signup

`page.tsx` (client component) → `POST /api/waitlist` → `lib/ratelimit.ts` (Upstash sliding-window, 5 req/60 s per IP, no-ops if env vars absent) → `lib/auth.ts` (`auth.api.signUpEmail`) → Neon Postgres via `lib/db.ts` (Drizzle + `postgres` driver, prepared statements disabled for pgBouncer).

Duplicate emails are silently treated as success to prevent enumeration. A hidden `company` honeypot field discards bot submissions server-side.

`GET /api/waitlist/count` queries the `user` table directly with Drizzle and returns the count for the live display on the landing page.

### Key libraries

- **Better Auth** (`lib/auth.ts`) — handles user creation, sessions, and rate limiting on `/api/auth/*`. GitHub OAuth activates automatically when `GITHUB_CLIENT_ID`/`GITHUB_CLIENT_SECRET` are set.
- **Drizzle ORM** (`lib/db.ts`, `lib/auth-schema.ts`) — schema is auto-generated by Better Auth CLI; don't hand-edit `auth-schema.ts`.
- **Upstash Ratelimit** (`lib/ratelimit.ts`) — guards `/api/waitlist` specifically, since that route calls `auth.api.signUpEmail()` server-side and bypasses Better Auth's own limiter.
- **Databuddy** (`app/databuddy-widget.tsx`) — analytics widget mounted after `{children}` in the root layout; isolated in a `'use client'` wrapper so the layout stays a server component.

### Styling

All styles live in `apps/web/app/globals.css` — plain CSS, no Tailwind. The design token for the brand orange is `--accent: #d97757`.

### Required env vars (`apps/web/.env.local`)

```
DATABASE_URL=          # Neon pooled connection string
BETTER_AUTH_SECRET=    # random secret, required
BETTER_AUTH_URL=       # canonical origin (e.g. https://burnusage.com), required in prod
UPSTASH_REDIS_REST_URL=    # optional; rate limiting degrades gracefully without it
UPSTASH_REDIS_REST_TOKEN=  # optional
GITHUB_CLIENT_ID=      # optional; OAuth activates automatically when set
GITHUB_CLIENT_SECRET=  # optional
```
